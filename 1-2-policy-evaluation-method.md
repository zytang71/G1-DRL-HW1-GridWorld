# 1-2 策略顯示與價值評估：預計實作方法

## 目標
- 在每個非終止狀態格子顯示一個隨機行動（上、下、左、右箭頭）作為初始策略 `pi(a|s)`。
- 根據固定策略做策略評估，計算每個狀態價值 `V(s)`。

## 狀態與動作定義
- 狀態 `S`：所有可通行格子（排除障礙物）。
- 終止狀態：使用者設定的終點（必要），起點只是起始位置不是終止。
- 動作 `A = {Up, Down, Left, Right}`。
- 邊界或撞到障礙物時，代理人維持在原狀態（`s' = s`）。

## 隨機策略顯示方式
- 對每個可通行且非終止狀態，等機率從四個動作抽一個顯示箭頭。
- 終點顯示 `G` 或特殊標記，不顯示箭頭。
- 障礙物顯示灰色，不參與策略與價值計算。

## 獎勵設計（建議）
- 每一步轉移獎勵 `r = -1`（鼓勵走最短路徑）。
- 轉移到終點給額外獎勵 `+10`（可調）。
- 折扣係數 `gamma` 建議預設 `0.9`。

## 策略評估演算法（Iterative Policy Evaluation）
1. 初始化所有可通行狀態 `V(s)=0`，終點固定 `V(goal)=0`。
2. 重複更新直到收斂：
   - `V_new(s) = sum_a pi(a|s) * sum_{s',r} p(s',r|s,a) * [r + gamma * V(s')]`
   - 若採「每格單一箭頭」的確定性策略，上式可簡化成：
     - `V_new(s) = r(s,a,s') + gamma * V(s')`
3. 計算 `delta = max_s |V_new(s)-V(s)|`，若 `delta < theta`（例如 `1e-4`）即停止。

## 預計整合到網頁的流程
1. 使用者完成地圖（n、起點、終點、障礙物）後按「產生隨機策略」。
2. 前端建立或向後端請求策略資料，顯示箭頭於格子內。
3. 執行策略評估迭代（前端 JS 或後端 Python 皆可）。
4. 於每格顯示 `V(s)` 數值（可保留到小數第 2~3 位）。

## API/程式結構建議
- `POST /api/generate-policy`：輸入地圖，回傳每個狀態動作。
- `POST /api/policy-evaluation`：輸入地圖、策略、`gamma`、`theta`，回傳 `V(s)` 與迭代次數。
- 前端維持單一來源狀態：`grid + policy + values`，避免畫面與資料不同步。

## 驗證重點
- 牆邊/障礙物碰撞的轉移是否正確（保持原地）。
- 終點是否維持固定值且不再更新。
- 不同 `gamma` 是否呈現合理價值變化（`gamma` 越高通常價值絕對值越大）。
